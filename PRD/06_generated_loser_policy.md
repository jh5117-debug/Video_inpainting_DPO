# Generated Loser Policy

## Old Multimodel Review

Reviewed files:

- `DPO_finetune/generate_multimodel_dpo_dataset.py`
- `tools/score_inpainting_quality.py`
- `DPO_finetune/Multimodel_DPO_Data_Generation_H20.md`
- `DPO_finetune/DPO_Dataset_Generation.md`

Reusable ideas from the old multimodel pipeline:

- Hard local masks already use area range, margin, static probability, slow motion, center jitter, and a constrained motion box.
- Candidates are generated as complete model outputs, then normalized and composited outside the mask with the GT/winner.
- Raw output and composited output are both kept under candidate folders.
- Cheap scoring already uses PSNR/SSIM, with optional LPIPS and VBench hooks.
- Candidate quality is converted to a relative quality score and bucketed into `too_good`, `too_bad`, and hard/balanced defect categories.
- Source balancing is already used when selecting two negatives, so one generation model does not dominate all DPO pairs.
- The old `neg_frames_1` and `neg_frames_2` idea maps cleanly to `loser_primary` and `loser_secondary`.

What should not be copied directly:

- The old source scanning is DAVIS/YouTube frame-directory oriented; VideoDPO must read `VIDEO_DPO_TRAIN_DATA_YAML` and preserve pair order.
- The old output schema targets `gt_frames`, `neg_frames_1`, and `neg_frames_2`; the new canonical pipeline must use JSONL manifests with raw/comp/final loser paths.
- The old relative score is computed only within a small old-style candidate set. The new v1 policy records absolute cheap metrics and a quality score in `[0, 1]`, then uses selection balancing.
- The old policy can stitch negatives by local segments. The new v1 keeps each candidate as a complete model output for a specific VideoDPO winner and mask.

## Canonical VideoDPO Setting

All generated-loser data in this phase must use the completed official VideoDPO / `official_videodpo_diffueraser` data setting:

- `source_dataset = videodpo`
- `train_data_yaml = $VIDEO_DPO_TRAIN_DATA_YAML`
- `num_frames = 16`
- `height = 320`
- `width = 512`
- `frame_stride = 1`
- `win = VideoDPO winner/chosen video`
- `prompt = VideoDPO prompt/caption`

The prompt is always preserved as source metadata. Text-conditioned generators
(`diffueraser`, `cococo`) receive it as model input; prompt-free generators
(`propainter`, `minimax_remover`) keep it in the manifest with
`prompt_used_by_model = false`.

Do not use arbitrary source resolution for generated loser data.

Current省时版 production data must set:

```text
generation_source = diffueraser_only
generation_model = diffueraser
```

Do not describe the active D2 partialmask K4 data as all-models source.

## Mask Policy V1

Policy name: `videodpo_partialmask_policy_v1_medium_hard_k4`

Config: `configs/generation/videodpo_partialmask_policy_v1_medium_hard_k4.yaml`

Fixed defaults:

| Parameter | Value |
| --- | --- |
| `num_masks_per_video` | `4` |
| `mask_area_min` | `0.20` |
| `mask_area_max` | `0.30` |
| `mask_margin_ratio` | `0.10` |
| `mask_static_prob` | `0.50` |
| `mask_speed_min` | `0.50` |
| `mask_speed_max` | `1.50` |
| `mask_center_jitter_ratio` | `0.04` |
| `mask_motion_box_ratio` | `0.16` |
| `mask_dilation_iter` | `0` |
| `mask_shape` | `irregular_polygon` |
| `mask_location` | `interior_constrained` |
| `mask_motion` | `two_static_two_slow_or_equivalent_random_with_static_prob` |

The first version targets medium-hard local perturbations:

- masks are not too small, so the loser is not nearly identical to the winner;
- masks are not too large, so DPO does not learn a trivial shortcut;
- masks stay away from image borders;
- mask motion stays inside a central motion box and does not sweep across the whole frame.

Every mask saves metadata:

- `mask_id`
- `seed`
- `area_ratio`
- `area_ratio_min`
- `area_ratio_max`
- `bbox`
- `bbox_ratio`
- `bbox_center_ratio`
- `bbox_margin_ratio`
- `motion_type`
- `velocity`
- `motion_center_bounds`
- `motion_box_ratio`
- `static_prob`
- `frame_level_bbox`
- `frame_level_area_ratio`

Implementation: `tools/videodpo_mask_policy.py`.

## Offline Multi-Mask Only

The first version is offline multi-mask generation, not online loser generation.

Reason:

- when the training mask changes online, the loser must be regenerated for that exact mask;
- otherwise `M_gen` and the modified loser region do not match;
- online generation is expensive, harder to reproduce, and adds training instability;
- offline K=4 masks per winner gives mask diversity while keeping the dataset stable.

For data-only experiments:

- `M_gen = offline sampled mask`
- `loser = offline generated by the inpainting model`
- `M_train = full mask`

For the later task partial-mask experiment:

- `M_train = M_gen`

## Candidate Generation

The original all-source design for partial-mask data was:

```text
K=4 masks x 4 generation models = up to 16 candidates
```

Candidate source models in that original design:

- `diffueraser`
- `propainter`
- `cococo`
- `minimax_remover`

Active 2026-05-25 production mode is DiffuEraser-only and supersedes the
all-source design for the current D1/D2 data:

```text
K=4 masks x 1 generation model = 4 candidates
generation_model = diffueraser
generation_source = diffueraser_only
```

This mode is chosen for throughput. It does not change the mask policy,
manifest schema, comp/no-comp path contract, or training adapter. It only
reduces the candidate source set so full data generation can complete before
launching DPO training.

Each candidate preserves:

- `raw_loser`
- `comp_loser`
- `mask`
- `generation_model`
- `mask_meta`
- `score_meta`
- `candidate_status`

Output layout:

```text

  candidates/
    <sample_id>/
      mask_000/
        mask/
        diffueraser/raw/

      mask_002/
      mask_003/
```

## Selection Policy V1

Policy name: `medium_hard_balanced_selection_v1`

Config: `configs/generation/medium_hard_balanced_selection_v1.yaml`

Required cheap metrics:

- `inside_mask_psnr`
- `inside_mask_ssim`
- `inside_mask_l1`
- `outside_mask_diff_mean`
- `outside_mask_diff_max`
- `temporal_diff`
- `frame_count`
- `height`
- `width`
- `decode_ok`

Optional metrics are disabled by default:

- `--enable_lpips false`
- `--enable_vbench false`

VBench can be used on calibration subsets only; do not run full VBench for all candidates by default.

Quality score:

- range: `[0, 1]`
- higher means the loser is closer to the winner / better
- `too_bad`: `quality_score < 0.30`
- `eligible`: `0.30 <= quality_score <= 0.65`
- `too_good`: `quality_score > 0.65`
- target: `0.45`

Primary loser:

- select an eligible candidate closest to target `0.45`;
- apply source balancing so no model dominates.

Secondary loser:

- select a second medium-hard candidate;
- prefer a different generation model;
- prefer a different defect bucket;
- otherwise choose the next closest candidate.

Fallback:

- if no candidate is in `[0.30, 0.65]`, choose the two candidates closest to target `0.45`;
- record `fallback_reason = no_candidate_in_quality_band`.

Implementation:

- scoring: `tools/videodpo_loser_candidate_scoring.py`
- selection: `tools/videodpo_loser_candidate_selection.py`
- calibration runner: `tools/videodpo_generated_loser_calibration.py`

## Source Balancing

First version source weights:

```yaml
diffueraser: 1.0
propainter: 1.0
cococo: 1.0
minimax_remover: 1.0
```

Selection ranking:

1. Prefer candidates in the quality band.
2. Prefer lower `source_pressure = selected_count[generation_model] / source_weight[generation_model]`.
3. Prefer smaller `abs(quality_score - 0.45)`.
4. Prefer secondary from a different generation model.
5. Prefer secondary from a different defect bucket.

Every selection writes `selection_meta` with source counts, source weights, quality order, selected primary/secondary, and fallback reason.

In DiffuEraser-only mode, source balancing is a no-op because all candidates
come from `diffueraser`. Selection still uses the quality band and target score
to choose primary and secondary candidates across the four masks.

## Primary And Secondary

Save both:

- `loser_primary`
- `loser_secondary`

First-version training uses only:

- `selected_primary_comp.jsonl` for partial-mask comp data;
- `selected_primary_fullmask.jsonl` for full-mask data.

`secondary` is retained for diagnosis, future multi-loser sampling, and future hard-negative ablations. Do not mix primary and secondary by default.

## Comp And No-Comp

Comp and no-comp must share the same selected candidate.

Partial comp:

```text
final_loser_video_path = comp_loser_video_path
```

Partial no-comp:

```text
final_loser_video_path = raw_loser_video_path
```

The selection score defaults to comp metrics because comp is the clean primary data-only experiment. No-comp manifests still keep raw metrics for diagnosing outside-mask changes.

Full-mask settings:

## Calibration Gate

Before full generation, run calibration:

```text
limit = 20 or 50 VideoDPO winners
models = diffueraser
partial masks K=4 for D2, full mask K=1 for D1
save all candidates
compute cheap metrics
select primary/secondary
save report
```

Report: `PRD/generated_loser_calibration_report.md`

Required report content:

- candidate count
- successful candidate count
- fail count by model
- selected model distribution
- quality score histogram
- too_bad / eligible / too_good ratios
- primary and secondary distributions
- mask area and motion distributions
- outside diff checks
- visual preview paths

Do not start full generation until calibration is reviewed.

PAI entrypoint:

```bash
python tools/videodpo_generated_loser_calibration.py \
  --output_root data/generated_losers/official_videodpo_diffueraser_data_partialmask_loser_k4 \
  --models diffueraser \
  --limit 20 \
  --mask_policy_config configs/generation/videodpo_partialmask_policy_v1_medium_hard_k4.yaml \
  --selection_config configs/generation/medium_hard_balanced_selection_v1.yaml \
  --calibration_report PRD/generated_loser_calibration_report.md
```

For the current DiffuEraser-only production pass, calibration/full-generation
commands should use:

```bash
--models diffueraser
```

Do not mix four-model pilot shards and DiffuEraser-only production shards in the
same `_shards` directory. Archive old `_shards` before restarting with a
different model set or worker/shard policy.
